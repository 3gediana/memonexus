# Memonexus - 面向AI Agent的长程记忆系统

## 一、问题与解决思路

### 1.1 核心问题

大模型 Agent 在多轮对话中面临**上下文记忆能力不足**的问题：
- 模型无法主动记住用户的历史偏好和事实
- 上下文 token 有限，无法将所有记忆都塞进去
- 传统方案依赖向量相似度召回，但**记忆之间的关联不是简单的语义相似，而是逻辑关联**

### 1.2 解决思路：双库架构

```
                    ┌─────────────────────────────────────┐
                    │          AI Agent 上下文             │
                    └──────────────┬──────────────────────┘
                                   │
                    ┌──────────────┼──────────────┐
                    ▼              ▼              ▼
             ┌──────────┐   ┌──────────┐   ┌──────────┐
             │  知识库   │   │  记忆库   │   │ 对话上下文 │
             │ (外置知识) │   │ (用户记忆) │   │  (当前轮)  │
             └──────────┘   └──────────┘   └──────────┘
```

**知识库**：模型的外置知识储备，用于模型缺乏专业知识时调用已准备好的资料，做专业回答

**记忆库**：用户的历史记忆，通过工具主动注入上下文，让模型在当前对话中参考用户历史记忆回答，回答完毕后记忆被"丢弃"（不参与下一轮对话），让模型专注于当下

### 1.3 核心创新

| 维度 | 传统方案 | Memonexus |
|------|----------|-----------|
| 存储触发 | 被动 | **主动筛选**（RoutingAgent） |
| 召回方式 | 向量相似度 | **逻辑关联**（图关系+动态权重） |
| 记忆结构 | 独立节点 | **记忆网络**（边强度+跨Key关联） |
| 记忆治理 | 无 | **事件驱动**（命中率+有效边+可见度） |
| 上下文控制 | 无 | **压缩Agent**（历史消息压缩） |

### 2.2 智能分块策略

传统分块脚本的痛点：每800-1000字强制切割，导致语义块被打断，质量差。

**Memonexus 智能分块优先级**：

1. **标题层级** - 按 h1/h2/h3 划分，保持文档结构完整
2. **代码块** - 保持代码块独立完整，不跨块切割
3. **段落** - 自然段落边界切割，保留语义完整性
4. **句子** - 最后兜底，按句子切分

### 2.3 用户输出格式（SANS）

当用户输入 `SANS(问题)` 格式时，触发特殊工具调用流程：

```
用户: SANS(关于RAG的原理是什么)
  │
  ▼
向量检索 Top-K 相关文档块（可能上千字）
  │
  ▼
本地小模型预处理（压缩为500字摘要）
  │
  ▼
注入模型上下文，回答用户问题
```

**优势**：既获取相关知识，又不占用过多 token

## 三、记忆库设计（核心）

### 3.1 四大Agent协作体系

```
用户消息
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│                  DialogueAgent（对话管理）                │
│  • 前台与用户对话，拥有记忆召回/存储/知识库工具            │
│  • 流式输出，支持多轮 tool-use 循环                      │
│  • 上下文超阈值时触发 CompressionAgent                   │
└─────────────────────────┬───────────────────────────────┘
                          │ 触发存储
                          ▼
┌─────────────────────────────────────────────────────────┐
│                   RoutingAgent（路由）                   │
│  • 筛选：判断消息是否值得记忆                            │
│  • 正则化：将"今天""明天"等相对时间转为绝对日期           │
│  • 分配：决定记忆归属哪个 Key                            │
└─────────────────────────┬───────────────────────────────┘
                          │ Key 分配结果
                          ▼
┌─────────────────────────────────────────────────────────┐
│                    KeyAgent（审核）                      │
│  • 决策四选一：                                          │
│    1. add - 新记忆，直接存储                            │
│    2. replace - 与已有记忆矛盾，更新                     │
│    3. duplicate - 与已有记忆重复，忽略                   │
│    4. reject - 不属于本Key，驳回重分                     │
│  • 去重检查：调用原文对比，防止假重复                    │
│  • 同Key建边：建立与已有记忆的关联                       │
│  • 重要性评分：0.1-0.9（关键/重要/普通/次要/琐碎）       │
└─────────────────────────┬───────────────────────────────┘
                          │ 主记忆 + 候选关联
                          ▼
┌─────────────────────────────────────────────────────────┐
│                AssociationAgent（跨域关联）               │
│  • 跨 Key 建边：将新记忆与其他 Key 下的记忆关联           │
│  • 关联强度：0.3（弱）/ 0.6（中）/ 0.9（强）           │
│  • 考虑"被同时召回"的场景                               │
└─────────────────────────────────────────────────────────┘
```

### 3.2 存储流程

```
1. 用户发送消息
2. DialogueAgent 生成回复（SSE流式推送思考过程）
3. 45秒静息计时器超时 或 用户开启新对话
4. FreezeManager 解冻，取出累积消息
5. 批量存储：
   ├─ RoutingAgent 筛选 + 分配 Key
   ├─ KeyAgent 决策（新增/替换/去重/驳回）
   ├─ KeyAgent 同Key建边
   └─ AssociationAgent 跨Key建边
6. SSE 推送存储结果
```

### 3.3 召回流程

```
1. 用户查询
2. DialogueAgent 解析意图，调用 recall_from_key
3. 遍历相关 Key（并行执行，每个Key独立召回）：
   ├─ 获取该 Key 下可见记忆（Top-N 可见度控制）
   └─ 原文+指纹 方式展示，每个 Key 最多8条
4. 跨 Key 扩展召回（共同召回的邻居记忆）
5. 组装上下文，模型生成融合记忆的回复
```

**并行召回设计**：多个 Key 同时召回，总时长控制在 4-5 秒内

### 3.4 对话压缩机制

当上下文 token 超阈值时，CompressionAgent 自动压缩历史消息：

- **用户消息**：保持完整
- **模型回复**：压缩为摘要，标记`（已压缩）`
- **效果**：上下文更清爽，模型更专注当前对话

## 四、记忆治理设计

### 4.1 核心洞察

**数字记忆不会过期**。传统记忆插件将记忆与时间绑定，但这是错误的——记忆作为信息本身，不会因为时间流逝而改变。

Memonexus 的记忆治理以**事件为驱动**，核心指标：

- **命中率（Hit Rate）**：记忆被召回后实际被模型用到
- **有效边强度（Effective Strength）**：边的命中率和边数量综合评估

### 4.2 命中率上报机制

```
模型生成回复
    │
    ├──► 回复中引用了某条记忆 ──► 命中 +1
    │
    └──► 回复中未引用某条记忆 ──► 未命中（不扣分）

命中事件上报后，EdgeCalibrator 动态调整边的有效强度
```

### 4.3 边强度动态校准公式

```
样本不足时（recall_count < 9）：
  calibration = 1.0（信任初始预判）

样本充足时：
  calibration = hit_count / recall_count（用实际命中率校准）

effective_strength = (1 - β) × 上次值 + β × (初始强度 × calibration)
其中 β = 0.2（学习率）

弹性约束：
  - 地板：不低于初始强度的 30%
  - 天花板：不超过初始强度
```

### 4.4 可见度机制（Top-N / Top-K）

**问题**：记忆库用久了会膨胀，不能无限制让模型看到所有记忆

**解决**：通过可见度（Visibility）控制记忆在模型眼中的出现频率

```
可见度更新公式：
  new_visibility = α × event_score × c_conn + (1 - α) × old_visibility

其中：
  - event_score：事件得分（关联召回=0.6，直接命中=1.0，直接未命中=0.2）
  - c_conn：连接度因子（边越多，衰减越慢）
  - α = 0.2

隐藏阈值：visibility < 0.3 时，记忆对模型不可见
```

**Top-N 可见度控制**：每个 Key 每次召回最多 N 条记忆，通过可见度排序筛选

### 4.5 记忆淘汰机制

当以下指标低于阈值时，记忆进入淘汰流程：

| 指标 | 说明 |
|------|------|
| 价值分（Value Score） | PageRank + ACT-R 模型计算 |
| 命中率 | 被命中的频率 |
| 边数量 | 关联的记忆越多越重要 |
| 有效边强度 | 关联的平均强度 |

**淘汰原则**：
1. 优先**隐藏**而非删除（隐藏的记忆可能通过关联被重新召回）
2. 删除条件严苛（需要所有指标都极低）
3. 参数**动态调整**，结合整体记忆库情况

### 4.6 价值评估模型

```
记忆价值 = PageRank权重 × PR值 + ACT-R权重 × ACT-R值 + 基础权重 × 初始评分 + 语义权重 × 语义有效性

其中：
- PageRank：记忆在关系网络中的重要性
- ACT-R：与学习率相关的记忆强度衰减模型
- 动态调整：随着记忆被使用，持续更新各维度权重
```

## 五、实时事件可视化

### 5.1 EventBus + SSE 架构

```
后端 Agent                           前端
    │                                  │
    ├─ emit_thinking() ─────────────►  │ Agent思考节点亮起
    ├─ emit_tool_call() ─────────────► │ 工具调用连线
    ├─ emit_result() ───────────────► │ 结果展示
    ├─ emit_storage_progress() ──────► │ 存储进度
    │                                  │
    │◄── EventBroadcaster（单例）───►  │ 广播到所有监控客户端
```

### 5.2 事件类型

| 事件 | 说明 |
|------|------|
| reasoning | 模型思考过程 |
| content | 流式回复内容 |
| tool_call | 工具调用触发 |
| agent_thinking | Agent思考中 |
| agent_tool_call | Agent工具调用 |
| agent_result | Agent执行结果 |
| storage_progress | 存储进度 |
| heartbeat | 连接保活 |

## 七、关键创新总结

1. **多Agent流水线**：Routing → Key → Association 三阶段，各司其职，专业化处理
2. **主动筛选机制**：不是被动存储，RoutingAgent 主动判断是否值得记忆
3. **逻辑关联而非向量**：记忆之间是图关系，通过边强度和跨Key关联建模
4. **事件驱动的记忆治理**：以命中率和有效边为核心指标，动态调整
5. **可见度控制膨胀**：Top-N + 动态可见度，防止记忆无限增长
6. **实时可视化**：EventBus + SSE，Agent协作过程全程可观测
